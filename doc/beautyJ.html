BeautyJ – Customizable Java Source Code Transformer
& Sourclet API with Java Source API

Normalizes Java source code to a clean structured format, customizable through the Sourclet API.
Auto-generates Javadoc comments using semantic information from identifier names.
Converts Java source code to XJava XML and back.
Contains the Java Source API to integrate Java source code parsing into any program.

SYNOPSIS

beautyj [-properties config-properties] -d output-dir
[-sourclet Sourclet-class] [-verbose] { Sourclet-options }
{ source-files-or-dirs } | -help

Description

BeautyJ parses Java source code and transforms it to different output formats. Source code can automatically be normalized to a clean structured format (“beautified”) or be converted to an XML representation for external transformation.

Javadoc-comments may be auto-generated, based on semantic information derived from identifier names. A standard project header can be attached to each file.

The output is fully customizable through the Sourclet API.

The Java Source API provides access to Java source code from any Java program, by parsing .java-files and representing their source code content in an object-oriented structure.

A usage example is given at the end of this document.

Running BeautyJ

There are several alternative ways to start up BeautyJ:

  1. Use the shell wrapper script beautyj.sh (on Unix-like systems) or beautyj.bat (on Windows). These scripts are located in the bin/ directory of the BeautyJ distribution. Example:
    beautyj -d=./src/beautified ...
    If you installed BeautyJ into a directory different from /usr/java/beautyJ-1_0 (on Linux systems) or c:\java\beautyJ-1_0 (on Windows systems), it might be necessary to manually adopt the paths inside the scripts to your configuration once before using BeautyJ for the first time.

  2. Assuming the classpath is already set to contain the BeautyJ classes (e.g. by including beautyj.jar in the classpath), start the Java interpreter with the wrapper class beautyj. This is placed in the root package of the classpath (that means, no leading package.subpackage... is required for calling it). Example:
    java beautyj -d=./src/beautified ...

  3. The beautyj.jar archive contains a manifest-file with information about the main class to invoke, so BeautyJ can be started up using the -jar parameter of the Java environment:
    java -jar [$BEAUTYJ_HOME/]beautyj.jar -d=./src/beautified ...

  4. BeautyJ can also be used as an ANT task to be included in ANT build scripts. See the ANT Task documentation below.

  5. Each of the above ways of invoking can make use of the '-gui'-option. Using this option, BeautyJ starts up with a graphical user interface, which allows you to edit all available options and start code beautification by hitting the 'Run'-button.
    Note that this is not the most sophisticated graphical user interface because it is automatically generated by the application framework BeautyJ is built on. However, it provides a fully graphical overview of all configurable options, which makes using BeautyJ much easier when you are not yet experienced in using it.

Options

Options of the StandardSourclet are either of type String, File, Class, Int or Boolean.

String, File, Class and Int options are specified using the syntax:

-option-name option-value

or

-option-name=option-value (the '=' is optional).

For example: -author.email me@somesite.com.

Note that when string option values contain spaces, system-dependent mechanisms to mark the blank character as part of the parameter need to be applied. For example, it is illegal to specify:

-project.name=This is my project

Instead, mark blank characters using

-project.name This\ is\ my\ project (Linux)

or

-project.name “This is my project” (Windows)

depending on your operating system.

Boolean options can be specified in two ways:

  1. Set the option directly by simply adding it to the list of parameters, for example (setting the option's value implicitly to true):

    -doc.method.description.param.create

  2. Boolean values can also be specified explicitly, e. g.

    -doc.method.description.param.create=yes
    -doc.fields.description.create=no

Either 'yes'/'no' or 'true'/'false' literals can be used to specify boolean values.

Javadoc auto-generation

Those options controlling the StandardSourclet's behaviour on how to handle Javadoc comments follow a common syntax pattern, which is:

-sourceObjectType.actionType.textType = { javadocElementType [,] }

The individual parts denote the following:

sourceObjectType := class | field | method

“class” refers to the single javadoc comment attached to the class definition, one per class

“field” associates the following parts of the option-name with field declaration of the class, that means declarations of variables and constants (final static variables in Java).

“method” refers to any method or constructor of the class.

actionType := create | remove

The “create”-action adds auto-generated documentation fragments to the Javadoc comment, if no such fragment already exists or an existing one has been previously removed.

By setting “remove”, documentation-fragments may be removed. Using “remove” and “create” together always creates auto-generated text on the element, while ommitting “remove” and using only “create” will keep existing comments and auto-adds a fragments only if it does not exist yet.

textType := empty | dummy | text

Empty documentation fragments can be referenced by specifying text-type “empty”.

All those text elements containing the substring “...” are considered to be incomplete dummies. By specifying ”dummy”, the StandardSourclet changes these dummy parts only. “Text” refers to any documentation text.

javadocElementType := description | author | version
(if sourceObjectType:=”class”)

or javadocElementType := description | param | throws | return
(if sourceObjectType:=”method”)

or javadocElementType := description
(if sourceObjectType:=”field”)

The Javadoc element type “description” refers to the text part of the Javadoc comment. “Author” and “version” are the corresponding Javadoc tags in the class documentation. For methods, handling the tags 'param', 'throws' and 'return' can also be specified.

Loading Options from a File

Default options can be loaded from a file instead of specifying them directly as parameters. A file that follows the simple Java Properties format (text lines with “key=value“ entries) can be specified using the paramater -properties filename. The values loaded from the file are treated as defaults, they can be overwritten by setting the option on the command-line.

-quiet [boolean]

(Default: false)

-properties [File]

The project's name.

-project.version

The project's version number.

-project.date

The project's date. (Default: today)

-project.description

Short description of the project.

-project.headerfile

A text file to insert as header comment (instead of auto-building a header with project and author information).

-author.name

The author's name.

-author.email

The author's email address.

-class.remove.dummy

Remove dummy content from a class-header's javadoc. Possible values are (multiple possible, comma-seperated): description, author, version

-class.remove.empty

Remove empty content from a class's javadoc comment. Possible values are (multiple possible, comma-seperated): description, author, version

-class.remove.text

Remove any content from a class's javadoc comment. Possible values are (multiple possible, comma-seperated): description, author, version

-class.create.dummy

Create dummy content for a class's javadoc comment. Possible values are (multiple possible, comma-seperated): description, author, version

-class.create.text

Create auto-text content for a class's javadoc comment. Possible values are (multiple possible, comma-seperated): description, author, version

-method.remove.dummy

Remove dummy content from a method's or constructor's javadoc comment. Possible values are (multiple possible, comma-seperated): description, param, return, throws

-method.remove.empty

Remove empty content from a method's or constructor's javadoc comment. Possible values are (multiple possible, comma-seperated): description, param, return, throws

-method.remove.text

Remove any content from a method's or constructor's javadoc comment. Possible values are (multiple possible, comma-seperated): description, param, return, throws

-method.create.dummy

Create dummy content for a method's or constructor's javadoc comment. Possible values are (multiple possible, comma-seperated): description, param, return, throws

-method.create.text

Create auto-text content for a method's or constructor's javadoc comment. Possible values are (multiple possible, comma-seperated): description, param, return, throws

-field.remove.dummy

Remove dummy description from a field's javadoc comment, if set to 'description'.

-field.remove.empty

Remove empty description from a field's javadoc comment, if set to 'description'.

-field.remove.text

Remove any description from a field's javadoc comment, if set to 'description'.

-field.create.dummy

Create dummy description for a field's javadoc comment, if set to 'description'.

-field.create.text

Create auto-text description for a field's javadoc comment, if set to 'description'.

-exception.texts

Sets default comment texts for thrown exceptions. The default values need not to be repeated if you want them leave unchanged and add some more. Example: 'MalformedURLException=if the url is not legal,NumberFormatException=if the string does not represent a number'."

-b or -code.braces.linebreak [boolean]

Choose between a code formatting style that outputs a line-break before the first starting curly brace ( '{' ) of a method (if set to true), or output it at the end of the method's declaring line (if set to false). (Default: false)

-c or -code.clean [boolean]

Remove out-commented dead lines of code from the inner method bodies. Every line that starts with '//' and ends with ';', '{' or '}' will be removed. (Default: false)

-code.separators [boolean]

Insert comment blocks between different types of class members. This highly increases readabilty. (Default: true)

-code.preserve.fields.order [boolean]

Preserve the order of field declarations as they are given in the orignal source file. In some cases this might be necessary to avoid illegal forward references when compiling the beautified sources. (Default: false)

-f or -code.format [boolean]

Format code blocks by auto-indenting the lines to a normalized standard-representation. (Default: false)

-i or -code.indent.spaces [int]

Sets the number of spaces with which auto-indentation gets performed. (Default: 4)

-q or -code.qualify [boolean]

Use fully qualified class names in the generated output. (E.g. use 'java.lang.String' instead of 'String'.) This might be needed in order to avoid ambiguity in class names. Usually, the classes and interfaces used in member signatures are declared by import statements. (Default: false)

-code.qualify.auto [boolean]

If option -code.qualify is disabled (which is default), fully qualifiy those class-names which would otherwise be unqualifiyable. (Default: true)

-xml.in [File]

Read Java source from XML file, not from .java-files. Then beautify and output as .java-files. Use "-xml.in=" without specifying a file name to read from standard in.

-xml.out [File]

Parse Java source and create XML file, without performing any beautification. Use "-xml.out=" without specifying a file name to write to standard out, in this case option -quiet is implicit.

-xml.validate [boolean]

Validate the input XML against its DTD before parsing. The input file must contain a <!DOCTYPE ..> declaration. Can be used only in combination with -xml.in.

-xml.doctype [boolean]

Include a <!DOCTYPE ..> reference into generated XML. Can be used only in combination with -xml.out.

Java Sourclets

A Sourclet generates a cleaned-up version of the code and (optionally) some auto-generated Javadoc information. The result is Java source code in a well-readable, normalized shape.

By default, a standard Sourclet is provided with BeautyJ which can be configured in a very flexible way and so is expected to serve many needs in reformatting Java source code. (Note that BeautyJ's standard Sourclet does not format the instruction-code inside method-bodies. In contrast, it cares about the overall organization of fields, constructors, methods and their Javadoc information inside a Java source code file.)

Developing your own Sourclets

A Sourclet is any Java class that implements the Sourclet-interface de.gulden.util.­javasource.sourclet.Sourclet . Any Sourclet can be passed to BeautyJ using the -sourclet parameter. By default, BeautyJ uses the Sourclet class de.gulden.util.­javasource.sourclet.standard.StandardSourclet, which covers a number of useful features for formatting Java source code.

To write your own Sourclet, you have three options:

  1. implement the interfaces de.gulden.util.­javasource.­sourclet.­Sourclet and de.gulden.util.­javasource.­sourclet.­SourcletOptions directly

  2. extend the base class de.gulden.util.­javasource.­sourclet.­AbstractSourclet, which provides pre-implemented abstraction of the code generation process.

  3. extend the existing standard Sourclet class de.gulden.util.­javasource.­sourclet.­standard.StandardSourclet to modify the features currently available

To introduce new option parameters for your Sourclet, register them in BeautyJ's internal configuration file application.xml which is placed in the package de.gulden.­application.­beautyj.

For any more details on the Sourclet API, refer to the Javadoc API documentation.

SOURCE FILES

Java source code

To parse .java-files as input (normal use), specify any number of directories or files as command-line parameters after the options.

BeautyJ processes syntactically valid, compilable Java source code files only. Any syntax error in class member declarations or Javadoc information will stop beautification. (However, errors inside method-bodies (the actual program code) should not confuse BeautyJ.) For more details about supported Java constructs and language versions see Limitations.

XJava XML

An XJava file contains a normalized representation of any number of packages, interfaces, classes, their members and the according javadoc documentation comments as XML. XJava files can be generated using option -xml.out, see Generated Files.

To read Java source from XJava XML, use option “-xml.in=file”. Specify “-xml.in=” (setting an empty parameter value) to read XJava from the standard input stream.

GENERATED FILES

Java source code

When beautifying existing code, the usual output format is .java source-files that can then be compiled or further edited.

The directory where to output the generated source files must be specified using option -d directory.

(Note that BeautyJ will rebuild one seperate .java-file for each class. If some of the original source files contain multiple classes in a single .java-file, the number of generated files will differ from the number of input files.)

XJava XML

Instead of beautifying code, BeautyJ can also generate an XML reperesentation of one or multiple Java source files. Such an XML file can be transformed externally in any possible way, and then be fed back to BeautyJ's beautification process using option -xml.input (see Source Files).

To create XJava output, use option “-xml.out file”. Specify “-xml.out=” (setting an empty parameter value) to write XJava to the standard output stream.

(Note that when XJava is created, no beautification will be performed. The Sourclet will not even be loaded by BeautyJ in this mode of operation.)

USAGE EXAMPLES

The following example parses the file /e/projects/beautyj/example/MyTestClass.java and outputs a beautified version into directory /e/projects/beautyj/example/beautified:

java -jar beautyj.jar "-project.name=This Is My Test Class" -project.version=0.8.15 "-author.name=Toby Tester"
-author.email=t.tester@somesite.com
-class.create.text description,author -method.create.text description,param,throws,return -field.create.text
-code.clean
-d=/e/projects/beautyj/example/beautified
/e/projects/beautyj/example/MyTestClass.java

The next example creates an XJava XML file from all sources recursively found in the ./src directory:

java -jar beautyj.jar -xml.out=/tmp/beautyJ.xml ./src

This runs BeautyJ with an simple graphical user interface, making it easy to get a complete overview of the availbale options:

java -jar beautyj.jar -gui

ENVIRONMENT

Command Line Shell

As a standalone application, BeautyJ is run from a command line shell by either directly invoking the main class de.gulden.application.beautyj.Main or the wrapper class beautyj. See Running BeautyJ.

ANT Task

Another way to run BeautyJ is to use it from inside an ANT build script. To declare the <beautyj ..> tag, include the following element once in the script:

<taskdef name=”beautyj”
class=”beautyjTask”
classpath=”beautyj_home/beautyj.jar”/>

Now BeautyJ can be used like any other ANT task inside a <target ..> .. </target> block:

<beautyj project_name=”testproject”
author_name=”Toby Tester”
class_create_text=”description,author”>
<src path=”/mydrive/project1/src”/>
<src path=”/anotherdrive/project2/src”/>
</beautyj>

Note that the parameter attributes inside the <beautyj>-tag are expected to be written with underscores ('_') instead of dots ('.'), so any option aaa.bbb.ccc becomes aaa_bbb_ccc in ANT.

For more information on how to use ANT, see the ANT documentation.

Java Virtual Machine

BeautyJ runs on a Java 2 platform, version 1.3 or higher. A Java Runtime Environment (JRE) is sufficient (JSDK is not required). It should be possible to also run BeautyJ in a Java 1.1.x environment, if the classpath additionally contains implementations of the new collection-classes (java.util.Collection, java.util.ArrayList, java.util.Iterator) introduced in Java 2.

Classpath

The classpath must provide access to the following packages and their sub-packages:

  • de.gulden.application.beautyj
    The BeautyJ main application classes.

  • de.gulden.util.javasource.sourclet
    The Sourclet API.

  • de.gulden.util.javasource
    The Java Source API classes.

  • de.gulden.framework.amoda
    The application framework classes used by BeautyJ, see AMODA – Abstract Modeled Application Framework.

  • org.w3c.dom
    The XML document object model (included in file lib/xmlParserAPIs.jar). Depending on the Java version, this package might already be part of the system classpath.

  • javax.xml.parsers
    SAX XML parser, implemented by Xerces 2.0 (included in file lib/xercesImpl.jar).

The distribution file beautyj.jar contains all packages required, so BeautyJ can be invoked by simply adding the option -jar <inst-dir>/beautyj.jar to the call of the Java interpreter.

When integrating BeautyJ into ANT, the ANT classes org.apache.tools.ant.* need to be accessible on the classpath (this is made sure by ANT, it is not necessary to specify the ANT classes as part of the classpath-attribute in the <taskdef ..> tag).

LIMITATIONS

Some limitations and bugs apply to the current version:

  • Line-indentation inside Javadoc-comments is not preserved. All lines will start at the left, no matter what their original indentation is (this may affect those Javadoc-comments that e. g. contain source code examples).

REUSABILITY

Java Source API

BeautyJ has been built on a reusable modular framework. The Sourclet API with the Java Source API can be applied as an independent library to access Java sources from any Java application. For detailled information, refer to the Javadoc API documentation of package de.gulden.util.javasource.

The Java Source API has been developed using JavaCC 2.0. The JavaCC source input files Parser.jjt and JavadocParser.jjt are included in the distribution. (Note: the JavaCC libraries are not required to run BeautyJ or the Java Source API, as JavaCC generates source files at compile-time and is not invoked at run-time.)

AMODA Application Framework

The BeautyJ application is developed on top of the AMODA – Abstract Modeled Application framework by the same author. AMODA provides cleanly modelled abstraction layers for options, commands, messages etc. that a program operates with. Additionally it controls the general lifecycle of an applicaton, for example by performing general initialization work. This way, AMODA dramatically reduces development efforts that have to be spent on small and medium sized applications (tools like BeautyJ, for example).

As AMODA distinguishes between the core application and the application environment in which it runs, a different environment can be “plugged” into the application without changing any code in the core application. BeautyJ demonstrates this mechanism by being runnable both as a standalone command-line application and as an ANT task without caring about this in BeautyJ's implementation.

BeautyJ's source codes serve as an example for developing new applications on top of AMODA.
