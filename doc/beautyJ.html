BeautyJ – Customizable Java Source Code Transformer
& Sourclet API with Java Source Code Parser API

Auto-generates Javadoc comments using semantic information from identifier names.
Converts Java source code to XJava XML and back.
Contains the Java Source Code Parser API to integrate Java source code parsing into any program.

SYNOPSIS

beautyj [-properties=config-properties] -d=output-dir [-sourclet=Sourclet-class] [-verbose] { Sourclet-options } { source-files-or-dirs } | -help

Description

BeautyJ parses Java source code and transforms it to different output formats. Source code can automatically be normalized to a clean structured format (“beautified”) or be converted to an XML representation for external transformation.

Javadoc-comments may be auto-generated, based on semantic information derived from identifier names. A standard project header can be attached to each file.

The output is fully customizable through the Sourclet API.

The Java Source Code Parser API provides access to Java source code from any Java program, by parsing .java-files and representing their content in an object-oriented structure.

A usage example is given at the end of this document.

Running BeautyJ

There are four alternative ways to start up BeautyJ:

  1. Use the shell wrapper script beautyj.sh (on Unix-like systems) or beautyj.bat (on Windows). These scripts are located in the bin/ directory of the BeautyJ distribution. Example:
    beautyj -d=./src/beautified ...
    If you installed BeautyJ into a directory different from /usr/java/beautyJ-1_0 (on Linux systems) or c:\java\beautyJ-1_0 (on Windows systems), it might be necessary to manually adopt the paths inside the scripts to your configuration once before using BeautyJ for the first time.

  2. Assuming the classpath is already set to contain the BeautyJ classes (e.g. by including beautyj.jar in the classpath), start the Java interpreter with the wrapper class beautyj. This is placed in the root package of the classpath (that means, no leading package.subpackage... is required for calling it). Example:
    java beautyj -d=./src/beautified ...

  3. The beautyj.jar archive contains a manifest-file with information about the main class to invoke, so BeautyJ can be started up using the -jar parameter of the Java environment:
    java -jar [$BEAUTYJ_HOME/]beautyj.jar -d=./src/beautified ...

  4. BeautyJ can also be used as an ANT task to be included in ANT build scripts. See the ANT Task documentation below.

Options

Options of the StandardSourclet are either of type string or type boolean.

String options are specified using the syntax

-option-name=option-value

for example, -author.email=me@somesite.com.

Note that when string option values contain spaces, system dependant mechanisms to mark the blank character as part of the parameter need to be applied. For example,

-project.name=This is my project

is illegal. Instead, mark blank characters using something like

-project.name=This\ is\ my\ project (Linux)

or

“-project.name=This is my project” (Windows)

depending on your operating system.

(Note: do not use -project.name=”This is...”, the quotation marks need to enclose the whole parameter definition.)

Boolean options can be specified in two ways:

  1. Set the option directly by simply adding it to the list of parameters, for example (setting the option's value implicitly to true):

    -doc.method.description.param.create

    Boolean values can also be specified explicitly, e. g.

    -doc.method.description.param.create=yes
    -doc.fields.description.create=no

  2. Alternatively, set several options at once by grouping them hierarchically, for example

    -doc.method=create

    has the same effect as setting the individual options

    -doc.method.description.create.dummy
    -doc.method.description.create.text
    -doc.method.throws.create.dummy
    -    doc.method.throws.create.text
    -    doc.method.param.create.dummy
    -    doc.method.param.create.text
    -    doc.method.return.create.dummy
    -    doc.method.return.create.text

    When hiearchically grouping boolean options, any option whose name starts with the left-hand-side of the equation sign and also contains the right-hand-side of the equation sign as parts of its qualified name, will be set to true.

    With this syntax, sophisticated configuration options can be expressed using only a small number of parameters on the command-line.

Boolean Option Syntax

Those boolean options controlling the StandardSourclet's behaviour on how to handle Javadoc comments follow a defined syntax pattern, which is

-doc .sourceObjectType .javadocElementType .actionType .textType

where the individual parts denote the following:

sourceObjectType := class | field | method

“class” refers to the single javadoc comment attached to the class definition, one per class

“field” associates the following parts of the option-name with field declaration of the class, that means declarations of variables and constants (final static variables in Java).

“method” refers to any method or constructor of the class.

javadocElementType := description | author | version
(if sourceObjectType:=”class”)

or

javadocElementType := description | param | throws | return
(if sourceObjectType:=”method”)

or

javadocElementType := description
(if sourceObjectType:=”field”)

The Javadoc element type “description” refers to the text part of the Javadoc comment. “Author” and “version” are the corresponding Javadoc tags in the class documentation. For methods, tags @param, @throws and @return can also be specified.

actionType := create | remove

“Create” adds auto-generated documentation fragments to the Javadoc comment, if no such fragment already exists or an existing one has been previously removed.

By setting “remove”, documentation-fragments may be removed. Using “remove” and “create” together always creates auto-generated text on the element, while ommitting “remove” and using only “create” will keep existing comments and auto-adds a fragments only if it does not exist yet.

textType := empty | dummy | text

Empty documentation fragments can be referenced by specifying text-type “empty”.

All those text elements containing the substring “...” are considered to be incomplete dummies. By specifying ”dummy”, the StandardSourclet changes these dummy parts only.

“Text” refers to any documentation text.

In most cases, one will use the hierarchic shortcut mechanism for boolean options (see above), and so choosing between creating dummy-comment-parts or text-comment-parts is usually skipped. Instead, specify for example -doc=remove.dummy -doc=create to let all dummy documentation fragments be removed and auto-generate any fragment that does not exist after clearing the dummies.

Loading Options from a File

Default options can be loaded from a file instead of specifying them directly as parameters. A file that follows the simple Java Properties format (text lines with “key=value“ entries) can be specified using the paramater -properties=filename. The values loaded from the file are treated as defaults, they can be overwritten by setting the option on the command-line.

Java Sourclets

A Sourclet generates a cleaned-up version of the code and (optionally) some auto-generated Javadoc information. The result is Java source code in a well-readable, normalized shape.

By default, a standard Sourclet is provided with BeautyJ which can be configured in a very flexible way and so is expected to serve many needs in reformatting Java source code. (Note that BeautyJ's standard Sourclet does not format the instruction-code inside method-bodies. In contrast, it cares about the overall organization of fields, constructors, methods and their Javadoc information inside a Java source code file.)

Developing your own Sourclets

A Sourclet is any Java class that implements the Sourclet-interface de.gulden.util.­javasource.sourclet.Sourclet . Any Sourclet can be passed to BeautyJ using the -sourclet parameter. By default, BeautyJ uses the Sourclet class de.gulden.util.­javasource.sourclet.standard.StandardSourclet, which covers a number of useful features for formatting Java source code.

To write your own Sourclet, you have three options:

  1. implement the interfaces de.gulden.util.­javasource.­sourclet.­Sourclet and de.gulden.util.­javasource.­sourclet.­SourcletOptions directly

  2. extend the base class de.gulden.util.­javasource.­sourclet.­AbstractSourclet, which provides pre-implemented abstraction of the code generation process.

  3. extend the existing standard Sourclet class de.gulden.util.­javasource.­sourclet.­standard.StandardSourclet to modify the features currently available

To introduce new option parameters for your Sourclet, register them in BeautyJ's internal configuration file application.xml which is placed in the package de.gulden.­application.­beautyj.

For any more details on the Sourclet API, refer to the Javadoc API documentation.

SOURCE FILES

Java source code

To parse .java-files as input (normal use), specify any number of directories or files as command-line parameters after the options.

BeautyJ processes syntactically valid, compilable Java source code files only. Any syntax error in class member declarations or Javadoc information will stop beautification. (However, errors inside method-bodies (the actual program code) should not confuse BeautyJ.) For more details about supported Java constructs and language versions see Limitations.

XJava XML

An XJava file contains a normalized representation of any number of packages, interfaces, classes, their members and the according javadoc documentation comments as XML. XJava files can be generated using option -xml.out, see Generated Files.

To read Java source from XJava XML, use option “-xml.in=file”. Specify “-xml.in=” to read XJava from the standard input stream.

GENERATED FILES

Java source code

When beautifying existing code, the usual output format is .java source-files that can then be compiled or further edited.

The directory where to output the generated source files must be specified using option -d=directory.

(Note that BeautyJ will rebuild one seperate .java-file for each class. If some of the original source files contain multiple classes in a single .java-file, the number of generated files will differ from the number of input files.)

XJava XML

Instead of beautifying code, BeautyJ can also generate an XML reperesentation of one or multiple Java source files. Such an XML file can be transformed externally in any possible way, and then be fed back to BeautyJ's beautification process using option -cml-input (see Source Files).

To create XJava output, use option “-xml.out=file”. Specify “-xml.out=” to write XJava to the standard output stream.

(Note that when XJava is created, no beautification will be performed. The Sourclet will not even be loaded by BeautyJ in this mode of operation.)

USAGE EXAMPLE

The following example parses file /e/projects/beautyj/example/MyTestClass.java and outputs a beautified version into directory /e/projects/beautyj/example/beautified:

java -jar beautyJ.jar "-project.name=This Is My Test Class" -project.version=0.8.15 "-author.name=Toby Tester" -author.email=t.tester@somesite.com -doc=create -code.clean -d=/e/projects/beautyj/example/beautified /e/projects/beautyj/example/MyTestClass.java

(Note that setting option -doc=create is a hierarchical shortcut for setting all of the following options:

-doc.class.description.create.dummy

-doc.class.description.create.text

-doc.class.author.create.dummy

-doc.class.author.create.text

-doc.class.version.create.dummy

-doc.class.version.create.text

-doc.method.description.create.dummy

-doc.method.description.create.text

-doc.method.param.create.dummy

-doc.method.param.create.text

-doc.method.return.create.dummy

-doc.method.return.create.text

-doc.method.throws.create.dummy

-doc.method.throws.create.text

-doc.field.description.create.dummy

-doc.field.description.create.text

Using this shortcut mechanism, it is also possible to set options in a more specific way, for example -doc.class=create -doc.method.param=remove.)

The next example creates an XJava XML file from all sources recursively found in the ./src directory:

java -jar beautyJ.jar -xml.out=/tmp/beautyJ.xml ./src

ENVIRONMENT

Command Line Shell

As a standalone application, BeautyJ is run from a command line shell by either directly invoking the main class de.gulden.application.beautyj.Main or the wrapper class beautyj. See Running BeautyJ.

ANT Task

Another way to run BeautyJ is to use it from inside an ANT build script. To declare the <beautyj ..> tag, include the following element once in the script:

<taskdef name=”beautyj”
class=”beautyjTask”
classpath=”beautyj_home/beautyj.jar”/>

Now BeautyJ can be used like any other ANT task inside a <target ..> .. </target> block:

<beautyj project.name=”testproject”
author.name=”Toby Tester”
doc=”create”>
<src path=”/mydrive/project1/src”/>
<src path=”/anotherdrive/project2/src”/>
</beautyj>

For more information on how to use ANT, see the ANT documentation.

Java Virtual Machine

BeautyJ runs on a Java 2 platform, version 1.3 or higher. A Java Runtime Environment (JRE) is sufficient (JSDK is not required). It should be possible to also run BeautyJ in a Java 1.1.x environment, if the classpath additionally contains implementations of the new collection-classes introduced in Java 2 (java.util.Collection, java.util.ArrayList, java.util.Iterator are used by BeautyJ's application framework AMODA).

Classpath

The classpath must provide access to the following packages and their sub-packages:

  • de.gulden.application.beautyj
    The BeautyJ main application classes.

  • de.gulden.util.javasource.sourclet
    The Sourclet API.

  • de.gulden.util.javasource
    The Java source code parser API classes.

  • de.gulden.framework.amoda
    The application framework classes used by BeautyJ, see AMODA – Abstract Modeled Application Framework.

  • org.w3c.dom
    The XML document object model (included in file lib/xmlParserAPIs.jar). Depending on the Java version, this package might already be part of the system classpath.

  • javax.xml.parsers
    SAX XML parser, implemented by Xerces 2.0 (included in file lib/xercesImpl.jar).

The distribution file beautyj.jar contains all packages required, so BeautyJ can be invoked by simply adding the option -jar <inst-dir>/beautyj.jar to the call of the Java interpreter.

When integrating BeautyJ into ANT, the ANT classes org.apache.tools.ant.* need to be accessible on the classpath (this is made sure by ANT, it is not necessary to specify the ANT classes as part of the classpath-attribute in the <taskdef ..> tag).

LIMITATIONS

Some limitations and bugs apply to the current version:

  • The Java source parser causes problems with constant Unicode-characters, like e. g. '\u0123'. For use with BeautyJ, use (char)123 as a workaround.

  • Java allows some strange freedom in declaring arrays. According to the Java language specification, Chapter 10.2, all three of the following declarations are valid and share the same semantics:
    1. public String[][] twoDimString;
    2. public String twoDimString[][];
    3. public String[] twoDimString[]; (!)
    It is not tested whether the Java source parser supports syntax 2) or 3), syntax 1) is recommended for use with BeautyJ.

  • Some uncommon Javadoc tags, especially those introduced after JDK 1.3, may cause the parser to fail. However, it should not be hard to extend the source code by the appropriate tag names. For more information on Javadoc tags supported by the current version of Sun's JSDK, see the Javadoc tool documentation.

REUSABILITY

Java Source Code Parser API

BeautyJ has been built on a reusable modular framework. The Sourclet API with the Java source code parser API can be applied as an independent library to access Java sources from any Java application. For detailled information, refer to the Javadoc API documentation of package de.gulden.util.javasource.

The Java source code parser API has been developed using JavaCC 2.0. The JavaCC source input files Parser.jjt and JavadocParser.jjt are included in the distribution. (Note: the JavaCC libraries are not required to run BeautyJ or the Java source code parser API, as JavaCC generates source files at compile-time and is not invoked at run-time.)

AMODA Application Framework

The BeautyJ application is developed on top of the AMODA – Abstract Modeled Application framework by the same author. AMODA provides cleanly modelled abstraction layers for options, commands, messages etc. that a program operates with. Additionally it controls the general lifecycle of an applicaton, for example by performing general initialization work. This way, AMODA dramatically reduces development efforts that have to be spent on small and medium sized applications (tools like BeautyJ, for example).

As AMODA distinguishes between the core application and the application environment in which it runs, a different environment can be “plugged” into the application without changing any code in the core application. BeautyJ demonstrates this mechanism by being runnable both as a standalone command-line application and as an ANT task without caring about this in BeautyJ's implementation.

BeautyJ's source codes serve as an example for developing new applications on top of AMODA.

FUTURE PLANS

  • It would be nice to see BeautyJ become a module for integration into NetBeans and other OpenSource development tools.

  • Formatting of inner method bodies should be improved. This could be achieved by incorporating other OpenSource auto-formatters' features into BeautyJ.

  • There should be more options similar to -doc.exception.text that allow specifying pre-defined text-fragments for auto-commenting.
